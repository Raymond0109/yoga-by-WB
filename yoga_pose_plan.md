| 项 | 决策 | 影响 |

## 1. 产品目标

基于 MediaPipe 检测人体体式 → 用**解剖骨骼模型（具体到每一块骨头）**渲染当前相对位置 → 对照**标准体式**的骨骼相对位姿与肌肉张紧状态 → 量化偏差并给出**可操作的矫正建议**，最终以 2D 叠加 + 3D avatar 双视图可视化呈现。

**用户价值一句话**：让用户"看见哪块骨头偏了、哪块肌肉紧张、具体该怎么调"。

## 4. 解剖骨骼与肌肉模型设计（核心难点）

### 4.1 为什么是难点
MediaPipe 只给 33 个关键点（肩、肘、腕、髋、膝、踝、脊柱少数点等），远少于"每一块骨头"。需要一套**推导规则**从 33 点重建出完整骨架。

### 4.2 程序化骨骼定义（skeleton spec）
在代码里用一份声明式规格描述骨骼树，例如（节选）：

```
颅骨(cranium)
 ├ 颈椎(C1-C7)      ← 锚定 nose/shoulder 间插值
 ├ 胸椎(T1-T12)     ← 锚定 肩→髋 脊柱线分段
 ├ 腰椎(L1-L5)
 ├ 骶骨/骨盆(pelvis) ← 锚定 left/right_hip 中点
 │   ├ 股骨(femur L/R)   ← 锚定 hip→knee
 │   ├ 髌骨(patella)
 │   ├ 胫骨/腓骨(tibia/fibula L/R) ← 锚定 knee→ankle
 │   └ 跖骨/趾骨
 ├ 锁骨(clavicle L/R) ← 肩点附近
 ├ 肩胛(scapula L/R)
 ├ 肱骨(humerus L/R)  ← 锚定 shoulder→elbow
 ├ 桡骨/尺骨(L/R)     ← 锚定 elbow→wrist
 └ 腕骨/掌骨/指骨(L/R)
```

- 每块骨含：名称、父子关系、锚定到的 landmark（或插值规则）、长度比例系数、渲染半径。
- 左右对称骨共享比例模板。

### 4.3 由 33 landmark 推导全骨
1. **锚定**：大关节（肩/肘/腕/髋/膝/踝）直接映射；脊柱关键段用肩-髋连线分段。
2. **比例模板**：以检测到的身高或 limb 长度归一化标准骨长比例（参照人体测量学），推导未直接检测的骨（指骨、椎骨、髌骨等）。
3. **解剖约束**：左右对称、骨长恒定、相邻骨端点必须重合（刚性连接）。
4. **置信过滤**：`visibility` 低的 landmark 用上一帧/模板补全，避免抖动。

### 4.4 肌肉模型（v2）
- 主要肌群（竖脊肌、腹直肌、腹斜肌、三角肌、肱二/三头肌、胸大肌、背阔肌、臀大肌、股四头肌、腘绳肌、小腿三头肌等）以**包绕对应骨的简化网格**表示。
- **张力估算（启发式，无真值需校验）**：`张力 = f(关节角相对标准的偏离, 肌肉长度变化, 拮抗肌关系)`。
- 颜色映射：冷色（蓝/绿=放松）→ 暖色（黄/红=紧张），附图例。

## 5. 标准体式库构建（Reference Library）

- **数据格式**：YAML/JSON，每体式含：
  - 名称、难度、关键关节目标角（如"左膝屈曲 170°""双臂上举与躯干夹角 15°"）
  - 骨相对位姿参考、参考肌肉张紧状态
  - 常见错误与对应矫正指令模板
- **首批覆盖**：10–15 个基础体式（山式、下犬、战士一二三、树式、三角式、桥式、婴儿式、板式、鹤禅、轮式等）。
- **来源策略**：
  1. 自建：基于瑜伽教材与解剖学定义关键角（可控、权威）。
  2. 录制工具：由教练/标准视频提取 landmark **自动生成参考**，大幅降低人工标注成本。
  3. 可选：引入公开 yoga pose 数据集做体式分类标签补充。

---

## 6. 分析算法（Analysis）

- **关节角计算**：三点法（父-关节-子），用世界坐标 xyz 计算空间角，避免纯 2D 透视畸变。
- **偏差度量**：
  - 角度误差 `Δθ = |θ_current − θ_ref|`
  - 骨端点位移误差（与标准 ghost 比对）
  - 左右对称性误差
- **评分**：加权平均 → 体式完成度 `Score%`。
- **矫正建议生成**：定位偏差最大的 1–3 个关节 → 输出可操作指令（如"左臂再上举约 10°""收紧核心、骨盆中立"），配 2D 箭头高亮 + 3D ghost 对照。

## 7. 分阶段实施计划（Roadmap）

### Phase 0 — 脚手架
- 项目结构、Python venv、依赖安装、输入抽象层（图像/视频/相机统一为帧迭代器）、FastAPI + 前端空壳。
- 交付：可运行 `python app.py`，浏览器打开本地页，能选摄像头/上传并看到原始帧。
- 验收：三源均能进帧迭代器并被前端显示。

### Phase 1 — 检测管线
- 接入 MediaPipe PoseLandmarker（IMAGE/VIDEO 模式），输出 33 landmark（image + world + visibility）。
- WebSocket 推流；2D canvas 叠加显示 33 点骨架连线。
- 交付：实时/视频骨架叠加。
- 验收：单人清晰场景 landmark 稳定，CPU 下 FPS ≥ 15。

### Phase 2 — 解剖骨骼渲染（2D）
- skeleton spec 声明 + landmark→全骨推导（锚定+比例+对称约束）。
- 2D 叠加渲染每块骨头（胶囊/圆柱），高亮偏差大的骨。
- 交付：2D 解剖骨骼视图（可显示每块骨）。
- 验收：脊柱/四肢/胸廓位置合理，偏差骨高亮正确。

### Phase 3 — 标准库 + 偏差分析 + 矫正建议（图像闭环）
- 标准体式库格式 + 首批 10–15 体式 + 录制工具。
- 偏差量化（角/位移/对称）→ 评分 → 矫正建议。
- 交付：选一体式 → 看偏差评分与文字/箭头建议。
- 验收：对明显错误体式能给出合理矫正建议。

### Phase 4 — 3D avatar 渲染
- Three.js 场景：程序化骨骼网格由推导骨位姿驱动。
- 2D 源画面 + 3D avatar 双视图并列；标准体式 ghost（半透明）叠加对照。
- 交付：混合 2D+3D 视图。
- 验收：3D 与 2D 基本对齐，可旋转，ghost 对照清晰。

### Phase 5 — 肌肉渲染与张力（v2）
- 主要肌群网格 + 张力启发式估算 + 冷暖着色 + 图例。
- 交付：肌肉图层开关 + 热力着色。
- 验收：承重体式主要肌显紧张、放松体式显放松，趋势合理。

### Phase 6 — 报告/导出/打磨（v2）
- 截图/PDF 报告（体式、评分、偏差图、建议）。
- UI 打磨、性能（GPU delegate）、摄像头权限与错误处理。
- 交付：可交付版本。

## 9. 风险与开放问题

| 风险 | 说明 | 缓解 |
|---|---|---|
| 33 点 → 全骨推导误差 | 远端骨（指/椎）靠模板推测 | 比例模板 + 可见度过滤 + 多帧平滑 |
| 单视角深度歧义 | 世界坐标 z 不够准 | 以角度为主、位移为辅；标注"估算" |
| 肌肉张力无真值 | 启发式需校验 | 先对照教材/教练标注做 sanity check |
| 标准体式主观性 | 不同流派标准不一 | 库内标注流派/难度，可扩展覆盖 |
| Mac 摄像头/GPU | 权限与 Metal delegate | 先 CPU 跑通，再开 GPU delegate |
| 实时 3D 性能 | 骨+肌网格量大 | v1 先 2D 闭环，3D 做轻量骨 |

### 11.2 已知限制（诚实标注）
- **肌肉张力是启发式**：`live = level × 匹配得分`，非真实肌电；未做真人标定。
- **标准库仅 12 体式**：缺倒立/手臂平衡/坐立体式等；选错体式会导致张力面板"看起来不对"（根因是目标肌群错配，非渲染 bug）。
- **2D 无法区分前后层肌肉**：如 biceps/triceps 只能叠在同一侧，靠 world-landmark 法线近似。
- **参考插画/卡通不适合做输入**：MediaPipe 对插画常返回 poses=0（检测不到人体）。
